## Context

当前 `MultiAgentGraph` 在两次去重完成后才统一调用 `process_documents` 进行 PDF 转换与切割，导致重复文档仍会进入处理链路，整体耗时偏长。现有流程在 `collect_second` 后一次性处理所有文档，且 `process_documents` 作为单一节点存在。

## Goals / Non-Goals

**Goals:**
- 在每次去重完成后立即处理文档，缩短无效等待时间。
- 将文档处理逻辑抽成类方法，供两个新节点复用。
- 保持向量化前的输入一致性（最终仍得到合并后的 `llama_docs`）。

**Non-Goals:**
- 不修改去重算法或文档处理的具体实现细节。
- 不调整 RAG 检索、问题池或答案生成的逻辑。
- 不引入新的外部依赖或存储结构。

## Decisions

- **新增两个处理节点并复用方法**：新增 `process_first_documents`、`process_second_documents` 节点，各自调用抽出的 `process_documents` 方法。
  - 替代方案：保留单节点但在节点内部按阶段处理；该方案依然延迟执行，不满足缩短等待时间的目标。
- **结果在状态中累积**：每个处理节点将其产物追加到 `llama_docs`，保证向量化节点只依赖一个聚合列表。
  - 替代方案：分别保存 `first_llama_docs`/`second_llama_docs` 再在向量化节点合并；需要更多状态字段与合并逻辑，复杂度更高。
- **保持屏障语义**：向量化节点继续等待 `init_vector_store` 和第二次处理节点完成，确保向量库与最新文档均就绪。

## Risks / Trade-offs

- **风险：重复处理仍可能发生** → 通过每次去重后立即处理，并在合并时避免重复追加。
- **风险：状态增长导致内存上升** → 只存储处理后的 `llama_docs` 聚合列表，不保留多份中间列表。
- **权衡：新增节点增加图复杂度** → 换取更早释放处理链路的收益与更清晰的阶段边界。
